# ControlIT Alpha Release Setup

Production-demo target: NetLock remains vendor-owned. ControlIT reads NetLock tables, writes only `controlit_*`, and runs with a least-privilege MySQL user after one privileged migration step.

## Fresh Install

Prerequisite: NetLock RMM and its MySQL database must already be installed, configured, and running. ControlIT setup does not install, configure, or start NetLock.

Alpha networking scope: this release is tested for self-hosted NetLock on the same host, Docker network, or local LAN. Public-domain deployments require a correctly configured reverse proxy/TLS layer plus matching ControlIT API URL and CORS origins. That hosted-domain path is not the current alpha target.

1. Generate local ControlIT environment:

```bash
./scripts/setup-controlit-env.sh
```

The script creates `.env`, generates `CONTROLIT_BOOTSTRAP_PASSWORD`, `CONTROLIT_JWT_SIGNING_KEY`, and `CONTROLIT_DB_PASSWORD`, then prints the bootstrap SuperAdmin email/password once. It does not generate or change NetLock credentials.

2. Fill remaining `.env` values:

```bash
MYSQL_ROOT_PASSWORD=<existing NetLock MySQL root password or migration user password>
MYSQL_DATABASE=iphbmh
MYSQL_CONTAINER=mysql-container
CONTROLIT_DB_HOST=mysql-container
CONTROLIT_DB_PORT=3306
NETLOCK_DOCKER_NETWORK=netlock-rmm-api-layer_netlock-network
CONTROLIT_NETLOCK_TOKEN=<remote_session_token from NetLock accounts table>
CONTROLIT_NETLOCK_FILES_KEY=<NetLock files_api_key>
CONTROLIT_NETLOCK_HUB_URL=http://netlock-rmm-server:7080/commandHub
CONTROLIT_PUBLIC_API_URL=http://localhost:5290
CONTROLIT_ALLOWED_ORIGINS=http://localhost:3000
NETBIRD_BASE_URL=<NetBird management URL>
NETBIRD_TOKEN=<NetBird personal access token>
```

LAN demo example:

```bash
CONTROLIT_PUBLIC_API_URL=http://<controlit-host-lan-ip>:5290
CONTROLIT_ALLOWED_ORIGINS=http://<controlit-host-lan-ip>:3000
```

Domain demo example:

```bash
CONTROLIT_PUBLIC_API_URL=https://api.<your-domain>
CONTROLIT_ALLOWED_ORIGINS=https://app.<your-domain>
```

3. Confirm existing NetLock/MySQL is healthy:

```bash
docker ps --format "table {{.Names}}\t{{.Status}}\t{{.Ports}}"
```

For external NetLock, verify equivalent MySQL and command hub reachability from the ControlIT host/network.

4. Run ControlIT EF migrations once with privileged MySQL credentials:

```bash
./scripts/run-controlit-migrations.sh
```

5. Create least-privilege ControlIT runtime DB user:

```bash
./scripts/apply-controlit-db-user.sh
```

6. Start ControlIT API + dashboard with runtime DB user:

```bash
docker compose -f docker-compose.controlit.yml up -d --build
```

7. Verify:

```bash
curl -f http://localhost:5290/healthz
curl -f http://localhost:3000
```

Login with the bootstrap email/password printed by setup. Change the password immediately.

## Existing NetBird Path

Use this path when the customer already has NetBird groups.

1. Set `NETBIRD_BASE_URL` and `NETBIRD_TOKEN` in `.env`.
2. Start ControlIT.
3. List groups as elevated admin:

```bash
curl -H "Authorization: Bearer <jwt>" http://localhost:5290/network/groups
```

4. Bind existing group to a tenant:

```bash
curl -X POST "http://localhost:5290/network/tenant-group?targetTenantId=<tenant_id>" \
  -H "Authorization: Bearer <jwt>" \
  -H "Content-Type: application/json" \
  -d '{"groupId":"<netbird_group_id>","mode":"external"}'
```

Use `mode:"read_only"` for visibility-only demos. Do not use `managed` here; managed groups are created internally by ControlIT.

## New NetBird Path

Use this path when ControlIT should create a tenant group/policy.

1. Set `NETBIRD_BASE_URL` and `NETBIRD_TOKEN` in `.env`.
2. Start ControlIT.
3. Create a tenant setup key through ControlIT:

```bash
curl -X POST "http://localhost:5290/network/setup-keys?targetTenantId=<tenant_id>" \
  -H "Authorization: Bearer <jwt>" \
  -H "Content-Type: application/json" \
  -d '{"name":"tenant-demo-key","type":"reusable","expiresInDays":7,"usageLimit":0,"ephemeral":false}'
```

ControlIT creates the managed NetBird group and isolation policy if missing. The raw setup key is returned only once.

## Device Install Pattern

NetLock agent install comes from the NetLock web console. NetBird install comes from NetBird setup key enrollment. Use customer-approved installer URLs and tenant-specific keys.

Linux pattern:

```bash
# NetLock agent - use exact command generated by NetLock web console for the target tenant.
curl -fsSL "<netlock_agent_installer_url>" -o netlock-agent.sh
sudo bash netlock-agent.sh --server "<netlock_server_url>" --tenant "<tenant_name_or_id>"

# NetBird agent - use tenant-specific setup key from ControlIT/NetBird.
curl -fsSL https://pkgs.netbird.io/install.sh | sh
sudo netbird up --management-url "<netbird_management_url>" --setup-key "<netbird_setup_key>"
```

Windows pattern:

```powershell
# NetLock agent - run tenant installer from NetLock web console.
powershell -ExecutionPolicy Bypass -File .\NetLock-Agent-Installer.ps1

# NetBird agent - install MSI/EXE from NetBird docs, then enroll with tenant key.
netbird up --management-url "<netbird_management_url>" --setup-key "<netbird_setup_key>"
```

## Required Environment

| Variable | Purpose |
|---|---|
| `MYSQL_ROOT_PASSWORD` | Existing NetLock MySQL root password or migration user password. ControlIT setup does not create it. |
| `MYSQL_DATABASE` | NetLock database, default `iphbmh`. |
| `MYSQL_CONTAINER` | Existing NetLock MySQL container name for local Docker user setup, default `mysql-container`. |
| `CONTROLIT_DB_HOST` | Existing NetLock MySQL host reachable from ControlIT containers. |
| `CONTROLIT_DB_PORT` | Existing NetLock MySQL port, default `3306`. |
| `NETLOCK_DOCKER_NETWORK` | Existing Docker network shared with NetLock, default `netlock-rmm-api-layer_netlock-network`. |
| `CONTROLIT_DB_USER` | Runtime DB user, default `controlit_api`. |
| `CONTROLIT_DB_PASSWORD` | Runtime DB user password. |
| `CONTROLIT_JWT_SIGNING_KEY` | HS256 signing key, at least 32 bytes. |
| `CONTROLIT_BOOTSTRAP_EMAIL` | Initial SuperAdmin email. |
| `CONTROLIT_BOOTSTRAP_PASSWORD` | Initial SuperAdmin password. |
| `CONTROLIT_NETLOCK_TOKEN` | NetLock `remote_session_token` for SignalR. |
| `CONTROLIT_NETLOCK_FILES_KEY` | NetLock `files_api_key` for admin REST status. |
| `CONTROLIT_NETLOCK_HUB_URL` | Existing NetLock SignalR command hub URL. |
| `CONTROLIT_PUBLIC_API_URL` | Browser-facing ControlIT API URL baked into the Next.js dashboard build. |
| `CONTROLIT_ALLOWED_ORIGINS` | Comma-separated browser origins allowed by ControlIT API CORS. |
| `NETBIRD_BASE_URL` | NetBird management URL. |
| `NETBIRD_TOKEN` | NetBird PAT. |
| `CONTROLIT_AUTO_MIGRATE` | Keep `false` in production/demo runtime. |

## Rotation

- Bootstrap password: change at first login; remove or rotate `CONTROLIT_BOOTSTRAP_PASSWORD` after a real admin exists.
- JWT signing key: rotate by replacing `CONTROLIT_JWT_SIGNING_KEY` and restarting API. Existing sessions become invalid.
- NetLock token: refresh after NetLock restart or admin token rotation; update `CONTROLIT_NETLOCK_TOKEN`, restart API.
- NetBird PAT: rotate in NetBird, update `NETBIRD_TOKEN`, restart API.
- NetBird setup keys: delete demo keys after enrollment; create tenant-specific short-lived keys for demos.
- Runtime DB password: alter MySQL user password, update `.env`, restart API.

## Security Notes

- Do not commit `.env`, tokens, setup keys, or generated helper output.
- Do not run ControlIT API as MySQL root after migrations.
- Do not edit NetLock source or NetLock-owned tables.
- External/read-only NetBird groups are customer-owned; ControlIT must not delete or overwrite them.
